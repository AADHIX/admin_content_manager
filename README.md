## Overview

**Admin Content Manager** is a Flutter application with two distinct roles:

| Role | Capabilities |
|---|---|
| **Admin** | Create, edit, and delete content cards (image URL + title + description) |
| **User** | Browse and read published content in real time |

Role assignment is handled via a hardcoded UID allowlist in `AuthService`, checked both client-side and enforced server-side by Firestore Security Rules. The `AuthGate` widget listens to the Firebase auth stream and routes each session to the correct dashboard automatically.

**Supported platforms:** Android · iOS · Web · Windows · macOS · Linux

### Key files

**`main.dart` — `AuthGate`**
Sits at the root of the widget tree. Subscribes to `AuthService.authStateChanges` and routes to `LoginScreen`, `AdminDashboardScreen`, or `UserDashboardScreen` based on auth state and role.

**`services/auth_service.dart`**
Wraps `FirebaseAuth`. Exposes `authStateChanges`, `signInWithEmailAndPassword`, `createUserWithEmailAndPassword`, `signOut`, and `isAdmin(uid)`. Admin UIDs are maintained in a static `_adminUIDs` list.

**`services/content_service.dart`**
All Firestore operations targeting `app_data/public_content/items`. Provides a real-time `getContentStream()` plus `addContent`, `updateContent`, and `deleteContent`.

**`models/content_item.dart`**
Plain Dart class with `fromFirestore` / `toMap` converters and a `copyWith` helper. Fields: `id`, `imageUrl`, `title`, `description`, `createdAt`.

## Firebase Setup

### 1. Create a project

Go to [Firebase Console](https://console.firebase.google.com) → **Add project**.

### 2. Enable services

- **Authentication** → Sign-in method → Email/Password ✓
- **Firestore Database** → Create database (start in test mode, then apply the security rules below)

### 3. Configure Flutter

```bash
flutterfire configure
```

This generates `lib/firebase_options.dart` and places `google-services.json` / `GoogleService-Info.plist` in the correct platform directories.

### 4. Deploy rules & indexes

```bash
firebase deploy --only firestore
```

## Admin Configuration

Admin access is controlled by a UID allowlist in `lib/services/auth_service.dart`:

```dart
static const List<String> _adminUIDs = [
  'XUNrXGBTcndZf0SpGGD5N1FccYj2',
  // Add more UIDs here
];
```

To grant admin access to a user:

1. The user signs up through the app first.
2. Find their UID: **Firebase Console → Authentication → Users**.
3. Add the UID to `_adminUIDs` in `auth_service.dart`.
4. Mirror the same UID inside the `isAdmin()` function in `firestore.rules`.
5. Redeploy rules: `firebase deploy --only firestore:rules`

> The client-side `isAdmin()` check controls UI routing. The Firestore rules are the authoritative enforcement layer — the two must be kept in sync.

## Firestore Security Rules

Content lives at `app_data/public_content/items/{itemId}`.

```javascript
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    function isAdmin() {
      return request.auth != null &&
             request.auth.uid in ['XUNrXGBTcndZf0SpGGD5N1FccYj2'];
    }

    function isAuthenticated() {
      return request.auth != null;
    }

    // Content items — read: any authenticated user; write: admin only
    match /app_data/public_content/items/{itemId} {
      allow read: if isAuthenticated();
      allow create: if isAdmin()
                    && request.resource.data.keys().hasAll(['imageUrl', 'title', 'description'])
                    && request.resource.data.imageUrl is string
                    && request.resource.data.title is string
                    && request.resource.data.description is string;
      allow update: if isAdmin();
      allow delete: if isAdmin();
    }

    // Parent container document
    match /app_data/{document} {
      allow read: if isAuthenticated();
      allow write: if isAdmin();
    }

    // Deny everything else
    match /{document=**} {
      allow read, write: if false;
    }
  }
}
```

## UI Design

The app uses a warm orange-on-cream palette throughout:

| Element | Color | Usage |
|---|---|---|
| Primary / accent | `#FF6B2C` | Header background, buttons, active field borders, icons |
| Background | `#FFF8F4` | Scaffold and card background |
| Surface | `#FFFFFF` | Login card, content cards |
| Muted text | `#9A7060` | Placeholders, secondary labels |

**Login screen** — split layout: large orange header with wave curve at the bottom, floating white card with rounded inputs, password visibility toggle, "Forgot Password?" link, and a "Create one" signup prompt.

**Admin Dashboard** — AppBar with an orange `ADMIN` badge chip + Logout button. Content form with three inputs (Image URL, Title, Description) and an "+ Add Content" button. Below the form, a live-updating list of content cards each showing the image, title, description, and inline edit (pencil) / delete (trash) icon buttons.

**User Dashboard (Content Hub)** — Same card layout as admin but read-only. Cards have an orange glow/shadow, a colored divider under the title, and description text in the muted brown tone. No CRUD controls are shown.

## Security

See [`SECURITY.md`](SECURITY.md) for the full policy. Key points:

- **Never commit `.env`** — all Firebase credentials belong in environment variables only.
- **Keep admin UIDs in sync** between `auth_service.dart` and `firestore.rules`.
- **Rotate API keys** regularly via Firebase Console and restrict each key to only the services it needs.
- **Schema validation** is enforced at the Firestore rules level — `create` rules verify that `imageUrl`, `title`, and `description` are all present, non-empty strings.
